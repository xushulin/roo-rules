## 问题追踪规则（全局）

你在完成**任何代码修改任务**（包括 Bug 修复、功能新增、重构、配置调整等）后，**必须**按以下流程维护项目中的 `plans/待解决问题清单.md`，无需再次请求批准。如果项目中尚不存在该文件，需先创建。

### 一、核心原则

`plans/待解决问题清单.md` 是项目**唯一权威**的待解决问题登记册。所有已知问题必须在此文档中有记录，所有已解决的问题必须从此文档中移除。

**排序规则**：清单按优先级从高到低排列（P0 → P1 → P2），同级按发现日期从旧到新。追加新条目时必须插入到对应优先级分组的末尾。

### 二、修改代码后的操作流程

每次代码修改完成后，按以下步骤操作：

#### 步骤 1：判断本次改动是否解决了某个已知问题

对照清单中各条目的描述和相关文件列表，判断本次改动是否修复了其中任何问题。

**如果修复了**：
- 将对应条目标记为"已解决"
- 将该条目从原优先级分组中**移除**，移动到文档末尾的"附录：已解决问题（归档）"章节
- 在归档条目中简要记录修复日期和修复方式

**如果没有直接修复但影响了某问题**：
- 在对应条目下追加更新说明（日期 + 变更简述）

#### 步骤 2：判断本次改动是否引入了新问题

自问以下问题：

- [ ] 本次改动是否引入了任何已知的未完成逻辑（如 TODO、FIXME、空实现、硬编码临时值）？
- [ ] 本次改动是否在某些场景下可能导致行为退化（如性能下降、准确率降低）？
- [ ] 本次改动是否留下了未处理的边界情况（如空输入、极值、异常路径）？
- [ ] 本次改动是否产生了新的技术债务（如代码重复、硬编码阈值、缺失错误处理）？
- [ ] 本次改动是否需要后续跟进工作（如文档更新、测试补充、性能优化）？

**如果以上任一问题答案为"是"**：在清单中追加新条目，**插入到对应优先级分组的末尾**。

新条目格式如下：

```markdown
### 问题标题

| 项目 | 说明 |
|------|------|
| **编号** | 分类-编号 |
| **优先级** | **P0（致命）/ P1（重要）/ P2（改进）** |
| **发现日期** | YYYY-MM-DD |
| **问题描述** | 简洁描述问题的现象、根因 |
| **影响** | 该问题对项目/用户的实际影响 |
| **相关文件** | 涉及的代码文件路径 |
| **相关计划** | 关联的 plans 文档（如有） |
| **解决方案** | 设想或计划中的解决思路（如有） |
| **状态** | 🔴 待解决 |
```

#### 步骤 3：更新维护日志

在文档末尾的"维护日志"表格中追加一行，记录本次操作（日期、操作类型、简要说明）。

### 三、编号规则（项目自定义）

各项目应根据自身模块划分定义编号前缀。以下为通用建议：

- 按功能模块或技术领域划分前缀（如 `API-`、`UI-`、`DB-`、`AUTH-`、`PERF-` 等）
- 编号按分类内递增，如 API-001、API-002…
- 如项目有特殊需求，可自定义编号方案，但必须在清单文档的附录中说明

### 四、优先级定义

| 优先级 | 含义 | 示例 |
|--------|------|------|
| **P0（致命）** | 直接导致功能不可用或严重影响核心业务 | 应用崩溃、数据丢失、核心功能完全不可用 |
| **P1（重要）** | 在特定场景下影响用户体验或数据准确性 | 部分功能异常、边界场景错误、性能明显下降 |
| **P2（改进）** | 代码质量、性能优化、边界情况处理 | 缺少注释、硬编码阈值、技术债务、文档落后 |

### 五、问题状态

| 状态标记 | 含义 |
|---------|------|
| 🔴 待解决 | 已识别但尚未开始处理 |
| 🟡 进行中 | 正在处理中 |
| 🟢 已解决 | 已修复，待下次代码修改时归档到"已解决问题" |
| 🟡 待确认 | 需验证当前代码是否已修复 |

### 六、禁止行为

- ❌ 修改代码后不检查问题清单是否需要更新
- ❌ 发现新问题但只在 commit message 中提及而不追加到清单
- ❌ 已解决的问题继续保留在清单的待解决分组中
- ❌ 归档已解决问题时不记录修复日期和方式
- ❌ 跳过维护日志更新
- ❌ 新问题追加到错误的优先级分组（P1 问题放到 P2 区域）
