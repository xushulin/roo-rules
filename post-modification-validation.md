## 代码修改后自检规则（全局）

你在完成**任何代码文件修改**后，**必须**立即执行最低限度的正确性验证，无需再次请求批准。

### 一、强制验证矩阵

| 文件类型 | 验证方式 | 通过标准 |
|---------|---------|---------|
| `.py` | `py_compile.compile()` 语法检查 | exit code 0，无 SyntaxError |
| `.json` | `json.load()` 解析验证 | exit code 0，无 JSONDecodeError |
| `.kt` / `.java` | 项目对应的 Gradle 编译 task（如 `compileDebugKotlin`） | BUILD SUCCESSFUL |
| `.xml`（Android） | 包含在 Kotlin 编译检查中 | 同上 |
| `.gradle.kts` | `gradlew help` 或同 etc价脚本语法验证 | BUILD SUCCESSFUL |

### 二、重命名/文本替换专项检查（强制）

当本次修改涉及**批量重命名**（变量名、键名、文件名、类名、函数名等）时，必须额外执行：

| 检查项 | 方式 | 通过标准 |
|--------|------|---------|
| 残留旧名 | 用 `search_files` 在相关目录搜索旧名称 | **零匹配**（或仅存在于注释/字符串中且确认无影响） |
| 新名覆盖 | 确认所有引用点都已更新 | 关键引用点（配置键、API 调用、文件路径）无遗漏 |

**典型场景**：将 `recommend` 重命名为 `hint` 后，必须在相关目录搜索 `recommend` 确认零残留。

### 三、执行时机

| 时机 | 必须执行 |
|------|---------|
| 每次修改 `.py`/`.json`/`.kt` 文件完成后 | ✅ 是 |
| 仅读取文件、仅计划文档修改 | ❌ 否 |

### 四、失败处理

- 语法验证失败 → **立即修复**，不得跳过进入下一步
- 残留旧名搜索到匹配 → **立即清理**，重新验证至零残留
- 编译失败 → 分析错误，**修复后重新编译**

### 五、禁止行为

- ❌ 修改 Python 文件后不运行语法检查就认为"改完了"
- ❌ 修改 JSON 配置文件后不验证合法性
- ❌ 批量重命名后不搜索残留旧名
- ❌ 编译失败后不修复就切换到其他任务
- ❌ 以"改动很小、不会有问题"为由跳过验证
- ❌ 等用户提醒后才补做验证（应主动执行）
