## 长时间命令输出重定向规范（全局）

你在通过 [`execute_command`]() 执行 CLI 命令时，**必须**遵守以下输出重定向规范，无需再次请求批准。本规则适用于所有项目、所有工作区、所有操作系统。

### 一、背景

Agent 在 [`execute_command`]() 执行期间处于阻塞状态，无法实时读取命令的中间输出。若命令执行时间较长且输出未重定向，可能出现以下问题：

- 终端缓冲区有限，早期输出被覆盖丢失
- 命令超时或崩溃后无回溯依据，Agent 只能盲猜失败原因
- 复杂输出（如编译错误列表、依赖解析树）无法被 Agent 事后精准分析

将输出重定向到临时日志文件能保证完整保留全部输出，供 Agent 在命令完成（或超时）后读取分析。

### 二、判定标准：何时必须重定向

对即将执行的每条命令，评估以下两个条件：

| 条件 | 判断依据 |
|------|---------|
| **预计耗时超过 30 秒** | 涉及网络下载、编译构建、模型训练/推理、批量数据处理、大数据集导入导出等 |
| **输出对后续决策关键** | 输出内容直接影响 Agent 下一步行动（如安装失败需定位缺失的包、构建错误需找到问题文件等） |

**两个条件同时满足 → 必须重定向**。仅满足其一 → 建议重定向，但不强制。两个都不满足 → 直接执行即可。

**典型必须重定向的场景**：

| 场景 | 示例命令 |
|------|---------|
| 依赖安装 | `pip install -r requirements.txt`、`npm install`、`conda install` |
| 项目构建 | `npm run build`、`mvn compile`、`cargo build` |
| 模型训练/推理 | `python train.py`、`python infer.py` |
| 批量数据处理 | `python batch_process.py`、数据库迁移脚本 |
| 长时间测试 | `pytest`（全量）|
| Git 大操作 | `git clone`（大仓库）、`git lfs pull` |

**不需要重定向的场景**：

| 场景 | 示例命令 |
|------|---------|
| 简单文件操作 | `dir`、`git status`、`ls` |
| 快速查询 | `python -c "print(...)"`、`git log --oneline -5` |
| 秒级完成的操作 | `mkdir`、`copy`、`move` |

### 三、重定向方式

#### 3.1 命令模板

在 [`execute_command`]() 的 `command` 参数中，将命令写为以下形式：

**Windows (cmd.exe)**：
```
(你的原始命令) > "temp\roo_cmd_<timestamp>.log" 2>&1
```

**macOS / Linux**：
```
(你的原始命令) > temp/roo_cmd_$(date +%s).log 2>&1
```

**PowerShell（跨平台）**：
```
(你的原始命令) > temp/roo_cmd_$(Get-Date -UFormat "%s").log 2>&1
```

`2>&1` 确保 stderr 和 stdout 都写入同一个日志文件，不会遗漏任何输出。

#### 3.2 日志文件命名约定

| 要素 | 说明 |
|------|------|
| **目录** | 统一使用项目根目录下的 `temp/` 子目录 |
| **前缀** | 统一使用 `roo_cmd_`，便于识别和批量清理 |
| **时间戳** | 使用精确到秒的时间戳，避免同名冲突 |
| **示例** | `temp/roo_cmd_1719400000.log` |

#### 3.3 多平台兼容写法

若不确定目标平台，可使用以下兼容写法：

```bash
# 跨平台 Python 一行生成日志路径（相对于项目根目录）
python -c "import time; print('temp/roo_cmd_' + str(int(time.time())) + '.log')"
```

### 四、命令完成后的操作流程

命令执行后（无论是正常完成还是超时），**必须**按以下步骤处理：

1. **检查 exit code**：从 [`execute_command`]() 返回结果中确认命令是否成功
2. **读取日志文件**：使用 [`read_file`]() 读取日志文件内容
3. **分析关键信息**：从中提取错误摘要、关键警告、进度状态等
4. **基于日志决策**：根据日志内容决定下一步行动（修复错误 / 继续 / 向用户报告）
5. **清理日志文件**：任务结束时删除临时日志，见第六节

### 五、超时处理

若命令设置了 `timeout` 参数且触发了超时：

1. 即使命令被中断，**已重定向的日志文件仍然保留**
2. Agent **必须**读取日志文件以了解命令在超时前做了什么、输出了什么
3. 基于日志内容判断是延长 timeout 重试还是改变策略

### 六、日志文件清理

临时日志文件属于"执行命令过程中产生的临时产物"，应遵循已有的 [`cleanup-temp-rules.md`](cleanup-temp-rules.md) 规则：

- 任务完成后**必须删除**不再需要的临时日志文件
- 若日志中包含有价值的诊断信息（如某次失败的完整堆栈），可在向用户报告后删除
- 禁止以"防万一"为由保留所有临时日志

清理命令示例：

**Windows**：
```
del temp\roo_cmd_*.log
```

**macOS / Linux**：
```
rm temp/roo_cmd_*.log
```

### 七、禁止行为

- ❌ 对预计超过 30 秒的关键命令不重定向输出，导致输出丢失
- ❌ 重定向后不读取日志文件就直接进行下一步决策
- ❌ 命令超时后不检查日志文件内容
- ❌ 任务完成后不清理临时日志文件
- ❌ 对简单秒级命令画蛇添足地重定向（增加不必要的文件 I/O）

### 八、自检清单

每次通过 [`execute_command`]() 执行命令前，自问：

- [ ] 该命令预计耗时是否超过 30 秒？
- [ ] 该命令的输出是否对后续决策至关重要？
- [ ] 若以上两者皆满足，是否已在命令中添加了重定向？
- [ ] 日志文件路径是否已使用项目 `temp/` 子目录？
- [ ] 任务结束时是否清理了临时日志文件？
