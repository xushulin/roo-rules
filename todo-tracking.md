## 待办事项追踪规则（全局）

你在完成**任何与项目计划相关的工作**（包括新建/更新 plans/ 设计文档、执行开发任务、完成改进优化等）后，**必须**按以下流程维护项目中的 `plans/待办事项清单.md`，无需再次请求批准。如果项目中尚不存在该文件，需先创建。

### 一、核心原则

`plans/待办事项清单.md` 是项目**唯一权威**的待办事项登记册。所有已知的开发任务、功能改进、重构计划必须在此文档中有记录，所有已完成的任务必须从此文档移至归档。

**排序规则**：清单按优先级从高到低排列（P0 → P1 → P2），同级按创建日期从旧到新。追加新事项时需插入到对应优先级分组的末尾。

### 二、与待解决问题清单的分工

两个清单**独立运行**，各自有独立的编号体系、生命周期和驱动方式。互链是**可选的**——不是每个问题都必然有对应待办，也不是每个待办都必然解决某个问题。

| 场景 | 记入清单 | 操作 |
|------|---------|------|
| 发现 Bug / 缺陷 / 不正确行为 | **待解决问题清单** | 按 issue-tracking 规则追加 |
| 规划新功能 / 性能优化 / 重构 | **待办事项清单** | 按本规则追加 |
| 问题分析后确定了执行方案 | **两个清单都更新（可选互链）** | 问题清单标注解决方案，待办清单拆解为具体任务，双向互链编号 |
| 执行任务时发现新问题 | **两个清单都更新** | 问题清单追加新条目，待办事项标注受阻状态 |
| 规划了纯新功能，不解决任何已知问题 | **仅待办事项清单** | 无需关联问题清单 |
| 发现了 Bug 但尚未确定修复方案 | **仅待解决问题清单** | 无需立即创建待办条目 |

### 三、触发更新的场景

| 场景 | 操作 |
|------|------|
| 新建或更新 plans/ 下的设计/计划文档 | 扫描文档中的执行步骤，同步追加到此清单 |
| 开始执行某个待办事项 | 状态更新为 🔧 进行中 |
| 完成某个待办事项 | 状态更新为 ✅ 已完成 |
| 定期整理（建议每次提交后检查） | 将 ✅ 已完成事项移至附录归档 |
| plans/ 文档被删除或废弃 | 检查关联事项，确认是否需取消 |
| 待解决问题清单有新增解决思路 | 拆解为具体待办事项，双向互链编号 |

### 四、编号规则（项目自定义）

各项目应根据自身模块划分定义编号前缀。以下为通用建议：

- 按任务类型划分前缀（如 `FEAT-` 新功能、`IMPROVE-` 改进优化、`DOC-` 文档、`TEST-` 测试、`INFRA-` 基础设施等）
- 编号按分类内递增，如 FEAT-001、FEAT-002…
- 如项目有特殊需求，可自定义编号方案，但必须在清单文档的附录中说明

### 五、编号互链约定（可选）

互链是**可选的**，仅在问题与待办之间存在明确的"解决"关系时才建立。一个待办可以解决多个问题，一个问题也可以拆成多个待办。

当需要互链时，在两个清单中互相引用：

- **待解决问题清单**中追加字段：`**对应待办**：FEAT-xxx / IMPROVE-xxx`
- **待办事项清单**中追加字段：`**解决问题**：OCR-xxx / DL-xxx`

**不需要互链的典型场景**：
| 场景 | 说明 |
|------|------|
| 纯新功能（如"牌局回放"） | 有待办，无对应问题 |
| 主动重构/技术升级（如"迁移 Kotlin 2.2"） | 有待办，无对应问题 |
| 刚发现但还没想好怎么修的 Bug | 有问题，无对应待办 |
| 文档整理、测试补充等日常维护 | 有待办，无对应问题 |

### 六、优先级定义

| 优先级 | 含义 | 示例 |
|--------|------|------|
| **P0（必做/阻塞）** | 阻塞其他任务、必须立即完成 | 核心功能缺失导致无法继续开发 |
| **P1（重要）** | 本迭代应完成的任务 | 计划中的功能改进、性能优化 |
| **P2（可选/改进）** | 锦上添花、有空再做 | 文档整理、代码美化、低优先级重构 |

### 七、待办事项状态

| 状态 | 图标 | 含义 |
|------|------|------|
| 计划中 | 📋 | 已规划但尚未开始 |
| 进行中 | 🔧 | 正在执行 |
| 已完成 | ✅ | 执行完毕，待归档 |
| 暂缓 | ⏸️ | 因依赖/优先级变化暂停 |
| 已取消 | ❌ | 不再需要执行 |

### 八、事项条目格式

新事项条目格式如下：

```markdown
### 事项标题

| 项目 | 说明 |
|------|------|
| **编号** | 分类-编号 |
| **优先级** | **P0（必做/阻塞）/ P1（重要）/ P2（可选/改进）** |
| **创建日期** | YYYY-MM-DD |
| **关联计划** | 链接到 plans/ 下的详细设计文档（如有） |
| **解决问题** | 关联的待解决问题编号（如 OCR-001），无则留空 |
| **预估工作量** | 小 / 中 / 大 |
| **前置依赖** | 依赖的其他待办事项编号，无则留空 |
| **状态** | 📋 计划中 |
```

### 九、禁止行为

- ❌ 仅在 plans/ 设计文档中列出步骤而不同步到待办事项清单
- ❌ 已完成事项长期保留在主清单中不归档
- ❌ 待办事项与问题清单条目互链后只更新一边
- ❌ 跳过维护日志更新
- ❌ 新事项追加到错误的优先级分组（P1 事项放到 P2 区域）
