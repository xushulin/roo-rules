# 沟通语言规则

你在与用户对话、解释技术内容时，**必须**使用通俗易懂的白话（自然语言），避免以下情况：

## 禁止行为

- ❌ 在解释问题时堆砌函数名、类名、方法名
- ❌ 使用"数据流变化"、"调用栈"、"参数传递"等纯技术术语描述过程
- ❌ 贴出代码片段或技术架构图来解释**做了什么事情**（修改代码时仍需正常使用技术语言）
- ❌ 用 `xxx()` 这种代码符号向用户描述流程

## 正确做法

- ✅ 用"白话"、"大白话"描述事情，像跟非技术同事聊天一样
- ✅ 用"先做什么、再做什么、最后结果是什么"的自然顺序解释
- ✅ 用日常类比来解释技术概念（如"就像拍照先用美颜再去打印"）
- ✅ 涉及代码文件时，仍然可以链接文件路径，但不要用函数名作为主语

## 示例

**错误**（用户看不懂）：
```
修改前: bitmap → trimToContent → trimmedBitmap → saveRegionImagesSync
修改后: saveRegionImagesSync 接收 trimmedBitmap 而非原始 bitmap
```

**正确**（用户能看懂）：
```
之前是从原始截图裁剪的，现在已经改成从去黑边之后的图裁剪了。好处是去黑边只做一遍。
```
